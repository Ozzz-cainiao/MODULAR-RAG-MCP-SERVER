## 2) 关键知识点

### 2.1 为什么要做“Markdown 结构友好”切分

- 纯长度切分会破坏标题语义和代码块完整性。
- RAG 检索时如果 chunk 结构被破坏，会降低召回质量与可解释性。
- 本实现优先保障两个结构：
  - 标题行（`#`/`##`）不被拆坏
  - 代码块（``` fenced block）不被打断

### 2.2 递归切分策略

- 先做一级 Markdown 分段（标题、代码块、段落）。
- 对普通段落按分隔符优先级递归切分（段落/换行/标点/空格）。
- 当找不到合适分隔符时再做固定长度强制切分。

### 2.3 overlap 设计

- 通过 `SPLITTER_CHUNK_OVERLAP` 在相邻 chunk 间保留少量上下文。
- 可降低“边界信息丢失”风险。
- 通过环境变量可调：
  - `SPLITTER_CHUNK_SIZE`
  - `SPLITTER_CHUNK_OVERLAP`

## 4) 本模块常见问题

### Q1：为什么不直接按固定长度切分？

固定长度虽然简单，但会经常切断语义单元（标题/句子/代码块）。
递归切分可以先尝试语义边界，再回退到固定长度，效果更稳。

### Q2：代码块为什么允许超过 chunk_size？

代码块被拆断会严重影响可读性和后续检索解释。
在结构完整性和长度约束冲突时，本实现优先保留代码块完整性。
