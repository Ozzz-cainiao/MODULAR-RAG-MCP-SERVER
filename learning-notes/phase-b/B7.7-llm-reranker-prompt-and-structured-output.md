## 2) 模块功能 / 流程位置 / 作用

### 2.1 模块功能

- 基于 LLM 对初步召回结果做语义重排。
- 读取 `config/prompts/rerank.txt` 构造提示词。
- 要求 LLM 返回严格 JSON：`{"ranked_ids": [...]}`。

### 2.2 在整体流程中的位置

- 位于 `retrieval` 之后、最终上下文组装之前。
- 典型链路：
  - query -> 检索召回（Top-K）
  - `LLMReranker` 重排
  - 选取更优候选喂给 LLM 生成答案

### 2.3 模块作用

- 提升候选排序质量，减少“召回但不最相关”的结果进入最终上下文。
- 提供结构化输出约束，降低下游解析不确定性。
- 在 LLM 调用失败时返回回退信号，保证链路可降级。

---

## 3) 关键设计点

- **结构化输出强校验**：
  - 必须是 JSON 对象；
  - 必须包含 `ranked_ids`；
  - 必须与候选集合完全一致且无重复。
- **失败回退机制**：
  - LLM 调用异常时不直接中断，返回原顺序候选；
  - 在 `metadata` 注入 `rerank_fallback=llm_failed` 作为回退信号。
- **Prompt 可注入**：
  - 支持通过 `RERANK_PROMPT_PATH` 注入测试模板，便于 mock。

## 5) 本模块常见问题

### Q1：为什么要求 LLM 只返回 JSON？

因为重排结果要进入程序化解析流程，非结构化文本会引入歧义。
固定 schema 可以把“解析风险”提前变成可测试的契约。

### Q2：为什么 schema 不合法要报错，而不是静默修复？

静默修复会掩盖模型输出质量问题。
在关键链路中采用 fail-fast，更利于定位和治理提示词/模型行为。
