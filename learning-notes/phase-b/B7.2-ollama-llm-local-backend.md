## 2) 关键知识点

### 2.1 为什么单独做 `OllamaLLM`

- Ollama 是本地模型服务，接口路径与云端 provider 不同。
- 独立实现可将本地后端差异封装在 provider 内部。
- 上层仍统一走 `BaseLLM.chat(messages)`，调用方式不变。

### 2.2 默认配置策略

- 默认 `base_url`：`http://localhost:11434`
- 默认 `model`：`llama3.1`
- 支持通过环境变量覆盖：`OLLAMA_BASE_URL`、`OLLAMA_MODEL`

### 2.3 错误处理策略

- 输入消息 shape 不合法时，抛出 `ValueError`（包含 provider 信息）。
- 连接失败/超时等传输异常统一包装为 `RuntimeError`。
- 错误信息保持可读，同时不泄露敏感配置字段。

## 4) 本模块常见问题

### Q1：本地 Ollama 和云端 LLM 在架构上有什么共性？

共性是都实现 `BaseLLM` 契约，通过 `LLMFactory` 统一接入。
差异被封装在 provider 内部，业务代码无需感知。

### Q2：为什么单元测试不用真实 Ollama 服务？

因为该层单元测试重点是“路由与错误处理逻辑”，而不是外部服务可用性。
使用 mock 可以更快、更稳定地覆盖成功与异常分支。
