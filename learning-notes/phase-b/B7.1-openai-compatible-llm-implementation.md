## 2) 关键知识点

### 2.1 什么是 OpenAI-Compatible 设计

- 多个厂商都提供兼容 OpenAI 风格的 Chat Completions 接口。
- 核心请求/响应 shape 接近，可复用输入校验与响应解析逻辑。
- 差异主要集中在 endpoint、认证头和部分路径参数。

### 2.2 为什么这里做 provider 拆分

- `OpenAILLM` / `AzureLLM` / `DeepSeekLLM` 各自封装配置差异。
- 上层只依赖 `BaseLLM.chat(messages)`，不感知具体厂商。
- 通过 `LLMFactory` + `settings.llm.provider` 实现无代码切换。

### 2.3 错误处理策略

- 输入消息 shape 不合法时，抛出 `ValueError` 并包含 provider 信息。
- 底层传输异常统一包装为 `RuntimeError`，并包含 provider + 错误类型。
- 响应结构不符合预期时，抛出可读错误，便于快速定位问题。

## 4) 本模块常见问题

### Q1：为什么这里强调“mock HTTP，不走真实网络”？

因为单元测试目标是验证“代码逻辑”和“接口契约”，不是验证外部服务可用性。

- 更快：本地秒级反馈；
- 更稳定：不受网络抖动和第三方限流影响；
- 更可控：可以构造成功/失败/异常等边界场景。

### Q2：OpenAI 与 Azure OpenAI 为什么要分开实现？

虽然都是 OpenAI-compatible，但 Azure 存在 deployment 路径与 `api-version` 等差异。
拆分后可读性更高，也方便后续单独扩展各 provider 的细节。
