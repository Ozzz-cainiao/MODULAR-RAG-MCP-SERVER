## 2) 关键知识点

### 2.1 为什么 Embedding 也做 provider 化

- 不同厂商的模型与计费不同，但上层只需要 `embed(texts)`。
- 统一 `BaseEmbedding` 契约后，业务侧可以无代码切换 provider。
- 与 LLM 层一致，保持工厂路由和错误处理风格统一。

### 2.2 输入校验与超长策略

- 空输入会直接报错，避免无意义调用。
- 超长输入支持策略化处理：
  - `EMBEDDING_OVERFLOW_STRATEGY=error`：抛出可读错误
  - `EMBEDDING_OVERFLOW_STRATEGY=truncate`：按上限截断
- 长度上限由 `EMBEDDING_MAX_TEXT_LENGTH` 控制（默认 8192）。

### 2.3 Azure 复用 OpenAI 核心逻辑

- Azure 实现复用 OpenAI 的输入校验与响应解析函数。
- 差异只保留在 endpoint 路径、`api-version` 与 `api-key` 头。
- 这样可降低重复代码，保持行为一致。

## 4) 本模块常见问题

### Q1：为什么超长文本不总是自动截断？

因为不同业务对“语义完整性”容忍度不同。
有些场景更希望 fail-fast（直接报错），有些场景更希望可用优先（截断后继续）。

### Q2：为什么 Azure 与 OpenAI 不直接合并成一个类？

两者虽兼容，但 Azure 的 endpoint 与认证方式不同。
拆成两个 provider 类，边界更清晰，也便于后续分别扩展。
