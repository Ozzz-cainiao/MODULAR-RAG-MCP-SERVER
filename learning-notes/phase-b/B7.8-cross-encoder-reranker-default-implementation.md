## 2) 模块功能 / 流程位置 / 作用

### 2.1 模块功能

- 对 Top-M 候选执行打分并重排。
- 支持 deterministic 打分流程（便于测试与回归）。
- 在打分失败/超时场景提供回退信号。

### 2.2 在整体流程中的位置

- 位于检索召回后、最终答案生成前。
- 与 `none`、`llm` 一样，属于 Reranker 插件层。
- 通过 `RerankerFactory` 按 `provider` 动态选择。

### 2.3 模块作用

- 提升候选排序质量，使更相关 chunk 更靠前。
- 通过 Top-M 控制重排范围，平衡效果与成本。
- 提供 `cross_encoder_failed` 回退信号，帮助 Core 层做稳定降级。

---

## 3) 关键设计点

- **Top-M 重排**：仅对前 M 个候选打分排序，尾部候选保持原顺序。
- **结构与契约校验**：
  - 校验 `query`、`candidates` 输入合法；
  - 打分数量必须与候选数量一致。
- **失败回退**：
  - 打分异常时返回原顺序；
  - 在 `metadata` 注入 `rerank_fallback=cross_encoder_failed`。

## 5) 本模块常见问题

### Q1：为什么只对 Top-M 重排，不对全部候选重排？

因为重排成本通常高于召回成本。
先召回再重排 Top-M 是常见工程折中，能在质量与性能间取得平衡。

### Q2：为什么失败时不直接抛错终止链路？

RAG 在线链路更看重可用性。
重排失败时返回回退信号并保留原顺序，可以保证服务连续性，同时保留诊断信息。
